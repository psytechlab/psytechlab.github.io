# Preparing Detective Pies
If you think about it, our platform is akin to a detective tool: the user tries to understand from the text whether the observed person wants to jump out the window or not. Our user must understand the context of the observed person's life and assess how bad everything is for them. Our models help with this by immediately highlighting "interesting" posts.

Okay, so our user has found a person on the edge, what's next? Deploy various psychotherapeutic judo to dissuade them from their deadly intention. And if the observed person is already stepping over the railing, how do you stop them? Call the police, friends, school, work, and so on. This is the only way.

But there's a nuance — you need to know who and where to call and about whom to report. The problem is that people don't often sign up with at least their real name, let alone address or workplace/school information. But there's hope: sometimes the observed individuals post information in their posts that can be used to identify them, up to phone numbers or bank cards. "Bank cards?" you might ask. Yes, it's simple: sometimes people have it so bad that they ask for money to be sent for food.

To avoid combing through all 100,500 posts looking for personal information, we developed a small service `kitoboy-pie` (personal information extraction). Its operation is extremely simple: you input text, and get labels with the type of personal information contained in the text as output. In its basic implementation, the service comes with a NER model from [slovnet](https://github.com/natasha/slovnet), which compared to other known solutions handles the basic set of named entities (persons, locations, organizations) quite well. However, it's important to remember that the model is primarily intended for the news domain. We're conducting an evaluation of how it works on our data right now.

Another basic component of our "pie" is an entity detector based on regular expressions. It's perfect for entities with limited variability. This includes mobile phones, bank cards, emails, and more. The component is managed by a config where regular expressions and their corresponding entities are written as a dictionary.

If you want to implement something of your own, you just need to wrap it in a descendant of the `AbstractIE` class and implement the `make_prediction` method, which should output a list of classes.

```python
class AbstractIE(InterfaceInformationExtractor):
    def predict(self, texts: str|list[str]):
        if isinstance(texts, str):
            texts = [texts]
        preds = []
        for t in texts:
            entites = self.make_prediction(t)
            entites.sort()
            preds.append(";".join(entites))
        return preds

        pass

```

Just don't forget to initialize the class in the main file afterwards.

To ensure service uniformity, we made it similar to Triton. However, we didn't implement everything like in Triton, of course. Just the necessary minimum that will allow integrating this service on par with real Triton services.

# Screenshots of the platform
Development of our platform is on the home stretch. We are starting to test basic use cases and write documentation. We decided to show you some screenshots.

This is what the data upload window will look like. Yes, for now it is assumed that uploading to the platform will be done through CSV files. No parsers are planned.

![](/assets/images/devlog_3_CreateAvatarNewPerson.png)

When adding data, you will be able to choose to create a new person's profile or attach to an existing one.

Another addition to the detective nature of our platform. A social media account belongs to some person, but one person can have multiple accounts on different social networks. And even on one social network, there can be any number, generally speaking. There is often a combination of a real and 'fake' account. Therefore, we have social media accounts - avatars, as we call them - and real persons separated from each other. At the same time, one person can have multiple avatars.

Moving forward. When you create several avatar pages, you will see a window like this, where you can visually track the progress of processing posts by models:

![](/assets/images/devlog_3_Main.png)

And this is what the avatar's posts window looks like:

![](/assets/images/devlog_3_Avatar.png)

This is where all the magic happens. When the models finish processing, the user can filter posts by assigned attributes - this is what model predictions are called within the platform. This is exactly how we will deliver an irreparable increase in volunteer efficiency.

To get additional insights, the user can look at a time chart where they can assess the intensity of signals at different points in time. This is especially useful if some account is put under monitoring. If the number of signals per unit time is growing, it means you need to monitor it more closely.
